**The Meta-Skill Recommendation: The One to Build**
→ If I had to pick one meta-skill that combines existing primitives into something that's never been fully executed on Stacks:
HODLMM Yield Router = hodlmm_get_bin_position (read, exists) + hodlmm_move_liquidity (write, now exists — shipped upstream as [hodlmm-move-liquidity](https://github.com/aibtcdev/skills/tree/main/hodlmm-move-liquidity) by @cliqueengagements) + zest_supply_asset (write, exists) + APY comparison logic. The loop: check if LP position is drifted → if yes, move liquidity to active bins → if HODLMM APY < Zest lending rate, withdraw from HODLMM and deposit into Zest → reverse when HODLMM bins are active again.

This is genuinely autonomous capital management. It earns fees or interest continuously. It has a clear decision tree you can implement in ~150 lines of TypeScript. It qualifies for the HODLMM bonus. And it produces something no existing skill does — a full round-trip from a drifted LP position back to yield-generating deployment.

Tier 1 — Build These First (Full Loop, HODLMM Bonus Eligible, High Signal)
#1 → [✅ SHIPPED UPSTREAM 2026-04-15] HODLMM Move-Liquidity Auto-Rebalancer (#2) — now live as [hodlmm-move-liquidity](https://github.com/aibtcdev/skills/tree/main/hodlmm-move-liquidity). Compose on top of it; don't rebuild. Original reasoning retained below for context on why composition-on-top is the play.
This was the highest-conviction pick for one reason: it was the only HODLMM write operation the SDK didn't wrap. Everyone who submitted a HODLMM read skill ran into the same wall — drift detection without execution. This skill closed that loop and was the natural "part 2" every existing HODLMM skill was missing. The raw contract call to the liquidity router was a meaningful technical lift. Now that it's shipped, the high-leverage move is to compose it into a full capital loop.

#2 → sBTC Yield Maximizer: DCA + HODLMM + Zest (#1)
Three protocols, one capital routing decision: HODLMM bins vs. Zest lending, go where APY is highest. This is the clearest expression of "agent as capital allocator." It also stacks the HODLMM bonus. The challenge is that it requires solid APY comparison logic and a clean decision function — if that's implemented well and evidenced in the .ts file, it will score in the 80s. Don't half-build this one; the APY routing decision needs to be real code, not a stub.

#3 → HODLMM Inventory Balancer — target-ratio drift correction. See [Issue #493](https://github.com/BitflowFinance/bff-skills/issues/493).
When a HODLMM position's token ratio drifts past a threshold, swap back to target via Bitflow. Unlike #2 which routes between protocols, this one keeps capital in HODLMM and maintains optimal inventory ratio. Full capital loop, HODLMM bonus eligible. Fits cleanly on top of the shipped hodlmm-move-liquidity primitive — call the CLI, don't reimplement.

Judging:
- Agent pre-judge scores all valid PRs out of 100 using static review by default and evaluates only the submitted SKILL.md, AGENT.md, and .ts files unless the operator explicitly instructs otherwise.
- Functionality & Correctness (25 pts max)
  - 20-25: Strong implementation evidence, clear subcommands, consistent argument handling, valid-looking JSON output patterns, and no obvious correctness or safety flaws.
  - 10-19: Mostly works but has notable inconsistencies, partial integration, weak validation, or suspicious gaps.
  - 0-9: Major correctness issues, severe doc/code mismatch, invalid output assumptions, or obviously unsafe behavior.
- Ecosystem Utility & Technical Leverage (25 pts max)
  - 20-25: High-leverage primitive or highly useful execution layer that clearly expands what AIBTC agents can reliably do. This includes DeFi execution, wallet and transaction primitives, identity and signing rails, payments, messaging, and reusable infrastructure.
  - 10-19: Useful but narrower, more incremental, or less reusable.
  - 0-9: Weak relevance, low leverage, or little clear value to AIBTC agents.
- Autonomy & Safety (20 pts max)
  - 15-20: Excellent AGENT.md with clear decision order, strong guardrails, spending or sizing limits, refusal conditions, error handling, cooldowns, and concrete autonomous safety logic.
  - 8-14: Basic guardrails present but incomplete, vague, or missing important controls.
  - 0-7: Missing or weak AGENT.md, no meaningful safety controls, or high risk of overspending, unsafe execution, or operational failure.
  - Any autonomous financial write skill without concrete spend/risk limits, refusal conditions, and operational safeguards in the reviewed evidence cannot score above 14/20 in this category.
- Documentation & Usability (15 pts max)
  - 12-15: Clear SKILL.md with complete YAML frontmatter using the metadata: nested format, good tags, accurate examples, and easy for other agents to discover and chain.
  - 6-11: Basic documentation, some missing pieces, or examples that are incomplete or slightly inconsistent.
  - 0-5: Poor, misleading, or incomplete SKILL.md.
  - NOTE: Submissions using flat frontmatter (tags as YAML arrays, boolean user-invocable, missing metadata: block) will score 0-5 in this category because they fail the AIBTC registry validator.
- Overall Impact (15 pts max)
  - 12-15: High real-world value — could materially help agents earn, spend, coordinate, deploy, or operate on Bitcoin/Stacks.
  - 6-11: Moderate potential.
  - 0-5: Low or no clear impact.
- Documented safeguards, limits, confirmations, or refusal conditions only receive full credit when they are implemented in the reviewed code. If they appear only in documentation, they are treated as guidance rather than implemented enforcement.
- If enforcement may exist elsewhere (for example in helper modules, backends, APIs, or smart contracts outside the reviewed files), judges should describe it as "not implemented in the reviewed files" or "not visible in the reviewed evidence set" rather than assuming it does not exist anywhere.
- Documentation/code mismatch is judged explicitly. Nonexistent flags, wrong argument formats, misleading outputs, or command examples that do not match the reviewed implementation should reduce both Functionality & Correctness and Documentation & Usability scores.
- Core infrastructure skills are fully eligible and should be judged on how much they expand agent capability, not only on whether they directly execute DeFi or yield strategies.
- Human panel (Bitflow + AIBTC): reviews top 10, shortlists top 5, does ranked-choice voting after real testing in AIBTC wallet.
